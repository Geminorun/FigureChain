## 总体架构准则

- 开始任何改动前，先阅读当前仓库结构、已有文档和相关代码，不凭空假设项目架构。
- 遵循当前项目已经采用的技术栈、目录约定和编码风格，禁止无必要地引入新框架、新运行时或新架构层。
- 保持模块边界清晰：入口层只做输入输出和编排，复杂逻辑放入专门模块或服务。
- 严禁在多个地方编写相同逻辑；可复用逻辑必须抽象并集中维护。
- 只实现当前任务所需能力，避免借机做无关重构、无关优化或过度设计。
- 修改公共接口、配置格式、数据结构或目录约定时，必须同步更新文档和测试。
- 对不确定的需求、风险较高的修改或可能破坏兼容性的改动，必须先澄清再实施。

## 目录职责规范

- 修改文件前先确认该目录的职责，新增文件必须放在职责匹配的位置。
- 配置、业务逻辑、测试、脚本、文档、数据资料和生成产物应分目录维护，不得混放。
- 入口文件保持轻量，只做参数解析、依赖组装和流程调度。
- 测试文件应与被测功能建立清晰对应关系，便于定位和维护。
- 文档应放入既有文档目录；没有文档目录时再创建。
- 原始资料、样例数据、生成产物和源码必须区分清楚，不得把业务实现写入资料目录。
- 大型本地数据、缓存、导入产物和临时文件默认不得提交，除非项目文档明确要求。

## 防止重复造轮子

- 新增函数、类型、组件、Prompt、Provider、脚本或配置项前，必须先检索仓库中是否已有等价实现。
- 已有公共工具、类型、测试 fixture 和配置读取逻辑应优先复用。
- 如果发现两个地方出现相似逻辑，应优先抽象复用，而不是继续复制。
- 不得为了单个用例硬编码答案；应抽象为可复用、可测试的能力。
- 引入第三方依赖前，先确认标准库或现有依赖不能合理解决问题。

## Python 与后端规范

- 优先使用当前项目指定的 Python 版本和依赖管理方式，不自行切换运行时或包管理工具。
- Python 项目依赖管理、脚本入口、格式化、lint、类型检查和测试命令必须遵循当前仓库约定。
- 公共函数、类方法、数据模型和任务入口必须添加清晰的类型注解。
- 无法确定类型时使用更窄的协议、抽象基类或 `Any` 的局部封装，禁止让 `Any` 在公共接口中扩散。
- 数据结构优先使用 `dataclass`、`TypedDict`、Pydantic model 或项目既有模型表达，不滥用裸字典传递复杂数据。
- Python 文件、函数和变量使用 `snake_case`；类使用 `PascalCase`；常量使用 `UPPER_SNAKE_CASE`。
- 文件名、类名、函数名、变量名应表达职责，避免含糊缩写。
- 代码注释只解释必要的业务边界、安全原因或复杂逻辑，避免重复代码本身。
- 不要写大而全的文件；文件变大时应按职责拆分。
- FastAPI 路由层应保持轻量，只负责请求解析、依赖注入、响应模型和错误映射；核心逻辑应放入 service、domain 或 repository 层。
- Pydantic schema 应区分 API 输入、API 输出、内部领域模型和数据库模型，避免一个模型跨越所有边界。
- 后台任务、批处理、导入和长耗时流程不得阻塞主请求；需要可观察的状态、日志和失败记录。

## 数据库规范

- PostgreSQL 连接必须通过环境变量或配置文件读取，严禁提交完整连接串、密码或本机固定路径。
- 数据库访问优先使用项目既有方式；如使用 SQLAlchemy，应保持模型、session 管理、事务边界和迁移脚本清晰分层。
- Alembic migration 必须可重复执行、可审查，并且只作用于明确的 schema 边界。
- 涉及数据导入、同步或批处理时，应优先设计为幂等流程；重复执行不得产生重复数据或破坏人工维护字段。
- 大批量写入应使用批处理、COPY 或项目认可的高效写入方式，避免逐行低效提交。
- 语义字段中的占位值、空值和异常值应统一归一化；原始数据如需保留，应放入明确的 raw/source 字段。
- 修改表结构、索引、约束、枚举或迁移策略时，必须同步更新文档和测试。
- 启用 `pgvector`、全文索引或扩展能力时，必须先明确数据规模、刷新策略、查询路径和回滚方案。

## 图数据库规范

- Neo4j 或其他图数据库应作为图查询、路径搜索或关系投影层使用，不应成为无法回溯的数据孤岛。
- 若 PostgreSQL 是事实源，图数据库中的节点和边必须能回溯到源表、源记录和同步批次。
- 图投影应有明确的重建机制；局部增量同步失败时，必须能通过全量重建恢复一致状态。
- 图查询结果不得绕过权限、可信度、审核状态或业务边界校验。
- 图模型变更必须同步更新投影脚本、查询代码、测试数据和说明文档。

## Next.js 与前端规范

- 遵循当前项目采用的 Next.js、React、TypeScript、样式系统和组件约定，不自行引入并行技术栈。
- 页面和组件应按职责拆分：数据加载、状态管理、展示组件、交互组件和 API 调用不要混在一个大文件里。
- TypeScript 类型必须从接口边界向内收敛，避免在组件间传递不透明的裸对象。
- 用户可见状态必须清晰区分 loading、empty、success、error 和 partial 状态。
- 表单、搜索、筛选和长列表交互应考虑防抖、分页、可取消请求和错误恢复。
- 前端不得硬编码密钥、内部连接串或只应存在于服务端的配置。
- UI 文案、字段名和持久化业务常量应分开维护，不得让展示语言泄漏到数据库常量或 API 协议中。
- 重要交互、布局变化和响应式行为应通过浏览器或截图验证。

## AI 与队列规范

- 所有模型调用必须经过统一 Provider 或客户端封装，禁止在业务代码中散落直接调用。
- Prompt 应集中管理，并配套输入输出约束；禁止在多个模块中复制大段 Prompt。
- 涉及结构化输出时，必须使用 schema、类型模型或等价方式校验，禁止直接信任模型返回。
- 模型输出只能作为待校验输入，不得绕过业务校验、安全校验、权限边界或人工审核流程。
- RAG、embedding、抽取、总结、候选生成等 AI 结果必须记录来源、输入版本、模型版本和生成状态。
- 长耗时任务应使用项目既有异步、队列或任务机制，禁止阻塞主请求流程。
- 任务失败必须有明确错误记录和可观察状态，禁止静默失败。
- 队列 payload 只传必要标识和参数，不传大对象、密钥或不可序列化对象。

## 安全与部署规范

- `.env`、生产配置、访问令牌、API Key、数据库密码和任何密钥严禁提交。
- 日志、错误信息、测试输出不得打印完整密钥或敏感数据。
- 所有外部输入必须校验，所有外部输出必须避免泄露无关字段。
- 文件路径、网络地址、模型名称、数据库连接等环境相关配置必须可配置，不得写死本机路径。
- 禁止执行用户提供的原始 SQL、Shell 命令或未经校验的文件操作。
- 涉及数据变更、迁移、删除、覆盖或批量操作时，必须确认影响范围并保留可回滚路径。
- 对大型本地数据文件执行删除、移动或覆盖前，必须确认路径位于当前工作区预期目录内。
- CI 或本地验证至少覆盖安装、格式检查、类型检查、测试和构建中的相关步骤。

## 文档语言偏好

- 工程文档默认使用中文。
- 需求分析、技术方案、实施计划、验收标准和决策记录使用中文。
- 代码标识符、命令、文件路径、API 字段名和必要英文技术术语保持原文。
- 外部英文资料可保留原文标题，但说明和结论应使用中文。
- 重要行为变化、配置变化、接口变化和测试方式变化必须同步更新文档。

## 提交信息语言偏好

- git commit message 默认使用中文。
- 保留 Conventional Commits 前缀，例如 `feat:`、`fix:`、`docs:`、`refactor:`、`test:`、`chore:`、`ci:`。
- 提交信息应说明实际改动，不写空泛描述。
- 示例：`docs: 更新项目协作规范`、`fix: 修复配置读取失败问题`、`test: 补充边界用例覆盖`。
